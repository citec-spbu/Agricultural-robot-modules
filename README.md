# Проект: Доработка модулей полевого робота-агроскаутера

## Описание проекта

Проект посвящён разработке и доработке мобильного робота-агроскаутера, который будет работать в аграрном секторе. Робот — это автономное устройство, предназначенное для выполнения мультиспектральной съёмки агрокультур, сбора данных о физико-химических свойствах почвы и других показателей с использованием различных сенсоров. Полученные данные передаются на сервер, где нейросетевые модели анализируют их и формируют рекомендации для агрономов.

### Основной процесс работы:
1. **Подключение и настройка робота**:
   После запуска клиентского приложения пользователь подключается к роботу и получает его спецификацию, которая включает данные о типах собираемых данных, конфигурации сенсоров и камеры. Спецификация может быть настроена вручную или автоматически в зависимости от конфигурации робота (например, если он оснащён только базовыми сенсорами или более сложной операционной системой).

2. **Запуск сессии и взаимодействие с БД**:
   При старте работы создаётся уникальная сессия для работы с роботом. Все данные, получаемые от робота (например, изображения, данные с датчиков), логируются в базе данных (БД), которая имеет гибкую структуру, поддерживающую быстрые изменения и добавление новых типов данных без необходимости затрагивать всю схему БД.

3. **Планирование маршрута и навигация**:
   Клиент может задать маршрут для робота, основываясь на заранее подготовленных данных о поле. Робот будет двигаться по заданным координатам, используя встроенные алгоритмы навигации и избегая препятствий, что гарантирует безопасное перемещение по полю.

4. **Сбор данных и обработка**:
   Во время движения робот собирает данные с датчиков (температура, влажность, pH почвы и другие параметры), а также снимает фотографии или видеоматерию с помощью камер. Эти данные передаются на сервер для дальнейшей обработки.

5. **Обработка данных с помощью ИИ**:
   Полученные данные от робота отправляются в ML-сервис, который использует нейросетевые модели для диагностики болезней культур, определения типов агрокультур, оценки состояния поля и других важных параметров. Результаты обработки формируются в виде рекомендаций для улучшения агротехнических процессов.

6. **Логирование и отчёты**:
   Вся информация о данных с робота, выполненных действиях и полученных результатах сохраняется в базе данных. Это позволяет не только отслеживать процесс работы, но и анализировать эффективность работы робота в долгосрочной перспективе.

7. **Модули системы**:
   - **Кибериммунная система** для защиты от кибератак.
   - **Модуль ИИ** для диагностики болезней пшеницы, картофеля и кукурузы, а также для классификации симптомов и оценки степени поражения.
   - **Модуль автоматического определения типов агрокультур** на основе изображений с камер.
   - **Алгоритм навигации и планирования траекторий**, учитывающий полевые условия и препятствия.

### Изменения в технических требованиях:
Изначально проект ориентировался на выполнение ряда конкретных задач, включая создание ИИ-модулей и работу с сенсорами. Однако требования были уточнены и изменены, что теперь включает следующие приоритетные задачи:

**Технические требования:**
- Спроектировать легко масштабируемую и легко изменяемую архитектуру приложения для удаленного управления роботом-агроскаутером, чтобы в дальнейшем можно было безболезненно дорабатывать и адаптировать проект под новые требования.
- Все модули должны быть выполнены с учётом модульности и возможности добавления новых функций и датчиков без необходимости переписывать основную часть системы.
- Требуется обеспечение гибкости работы с базой данных (БД), где возможны изменения в структуре данных и добавление новых данных, таких как новые сенсоры или изменения в конфигурации робота.

Система должна обеспечивать следующие требования:
1. Поддержка многослойной архитектуры.
2. Использование нейросетевых фреймворков TensorFlow и PyTorch для реализации ИИ-алгоритмов.
3. Интеграция с системой ROS 2 для реализации высокоуровневого управления и связи между компонентами робота.
4. Совместимость с операционной системой Ubuntu/Astra Linux для обеспечения удобства разработки и работы в средах с открытым исходным кодом.

## Технический стек:
- **Языки программирования**: C++, Python
- **Фреймворки и библиотеки**: TensorFlow, PyTorch, ROS 2, QT
- **Операционная система**: Ubuntu, Astra Linux

### Ожидаемый результат:
Проект должен завершиться созданием приложения для функционирования с роботом, который автономно выполняет задачи в аграрном секторе, собирает данные с различных сенсоров и передаёт их на сервер для дальнейшего анализа и рекомендаций. Важным аспектом является создание инфраструктуры для масштабируемости и легкости в дальнейшем расширении проекта.

## Структура репозитория и описание веток

### AgroDesktop
Основная интеграционная ветка проекта.
Содержит десктопное приложение, объединяющее:

- ML-сервис обработки данных,

- сервис имитации робота,

- основное пользовательское приложение для работы с картой поля.

Ветка предназначена для демонстрации полного сквозного сценария работы агроскаутера: от управления роботом и навигации до визуализации данных и результатов ИИ-анализа.

### Map
Ветка с реализацией виджета для работы с картой (приложение **GeoViewMap**).
Используется для:

- визуализации полей и маршрутов движения робота,
- отображения координат, точек сбора данных и результатов анализа,
- интеграции с библиотекой QGeoView.

**Структура модуля:**

- **GeoViewWidget** — основной виджет-оркестратор: карта, слои, контур, робот, подключение обработчиков.
- **GeoViewDrawing** — рисование на карте: маршруты (GeoPolyline), иконки, очистка слоёв, загрузка изображений.
- **GeoViewToolbar** — панель управления: выпадающие кнопки (меню) и информационный список; при выборе пункта испускает сигналы, обработка в виджете.
- **GeoViewRouteLogic** — логика маршрутов: пересечения сегментов, проверка точки внутри полигона (ray casting), параллельные линии, упорядочивание точек (жадный алгоритм + 2-opt), генерация JSON-команд (rotate/move), гео-расчёты (haversine, азимут, Gazebo).

**Функциональность:**

- **Разместить робота** (меню): «Указать на карте» — клик по карте; «Из конфига» — выбор JSON-файла с полями `latitude`, `longitude` (опционально `rotation_angle`). После загрузки из конфига карта перемещается к позиции робота.
- **Добавить контур (GeoJSON)** — загрузка контура поля; карта подстраивается под границы контура.
- **Построить параллельный маршрут** (меню): «Автоматически» — подбор угла по близости к роботу; «Угол вручную» — ввод угла в градусах. Требуется предварительно разместить робота.
- **Построить команды** (меню): «Команды (rotate, move)» — вывод JSON в список Info; «Точки маршрута (широта, долгота)» — вывод списка координат в Info.
- **Указать маршрут вручную** (меню): «Кратчайший путь», «По порядку указания», «Завершить построение». При добавлении точек проверяется нахождение клика внутри контура (если контур задан).
- **Экспорт маршрута** (меню): «JSON для Gazebo» — экспорт с опциональным контуром и командами; «Список координат» — сохранение в файл строк «широта, долгота».
- При смене позиции робота маршрут пересчитывается (параллельный — заново, ручной — обновление первой точки и при необходимости переупорядочивание).

Данная ветка может использоваться как независимый модуль в других частях проекта.

### Wheat-weed-detection-model
Ветка, содержащая полный ML-пайплайн по обнаружению пшеницы и сорняков:

- конвертация разметки исходных данных,

- подготовка датасета в формате YOLO,

- обучение модели,

- валидация и инференс.

Итоговая модель обучена на YOLOv8 и демонстрирует стабильные результаты даже на ограниченном объёме данных.

### gazebo
Проект для управления роботом в симуляционной среде
ROS 2 Humble + Gazebo.

Реализует:

- управление движением робота,

- приём управляющих команд в формате JSON,

- тестирование алгоритмов навигации и управления без использования реального оборудования.

### manager-and-db
Ветка с реализацией:

- модулей работы с базой данных,

- внутреннего менеджера системы.

Отвечает за хранение, структурирование и доступ к данным, получаемым от робота, а также за управление сессиями и состоянием системы.

### ml_service
🚀 Асинхронный ML-сервис на базе FastAPI и Redis.

Сервис обрабатывает задачи через очередь и поддерживает:

- детекцию объектов (/detect/),

- прогноз рекомендаций по удобрениям (/fertilizer/).

Используется как независимый ML-бэкенд, с которым взаимодействуют клиентское приложение.

### network
Ветка с реализацией модуля сетевого взаимодействия.
Отвечает за:

- обмен данными между клиентом, сервером и роботом,

- передачу телеметрии, команд управления и результатов обработки,

- обеспечение устойчивого сетевого взаимодействия между компонентами системы.

### qt_version
Ветка с реализацией базы данных и связанных компонентов с использованием Qt.
Предназначена для тестирования и интеграции БД в десктопное приложение.

### robot_sim_testing
Сервис-имитация робота для тестирования логики управления.

Поддерживает:

- движение по координатам,

- вращение и остановку,

- отправку данных о текущем состоянии робота.

Реализован на FastAPI и предоставляет REST API для управления и мониторинга состояния робота.

### robot_sim
Начальная версия сервиса-симулятора робота.

Используется для:

- тестирования навигационных алгоритмов,

- сбора телеметрии,

- обработки данных от сенсоров.

Сервис учитывает спецификацию датчиков и передаёт данные в бинарном формате, что приближает поведение симулятора к реальному устройству.
Реализован на базе FastAPI.

### weed_segmentation
Ветка, посвящённая задаче семантической сегментации сорняков на изображениях пшеницы.
Используется для более точного анализа состояния посевов и оценки засорённости поля.

### wheat_disease_classification
Модель классификации болезней листьев пшеницы на основе ResNet18.
Предназначена для автоматической диагностики заболеваний по изображениям, полученным с камеры робота.

### wheat
Ветка с данными и экспериментами для сегментации пшеницы.

Содержит:

- обучающий, валидационный и тестовый датасеты,

- скрипты обучения и инференса,

- расчёт метрик качества.

Модель сегментации построена на U-Net с энкодером ResNet34.
